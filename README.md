# Fireeeee — 火災求生模擬器（有系統 vs 無系統）

## 🆕 最新更新 (2026-07-13)

### 量化指標補齊為完整 8 項

原本只有「逃生率」「進入高煙區比例」「臨界CO暴露比例」三項，現在加入五項與「系統決策品質/資訊即時性」直接相關的指標：

- **平均逃脫所需時間**：只計算最終**成功逃脫**的人，從起火到實際逃出當下的 tick 數取平均。死亡或模擬結束仍受困的人不計入這項平均值（避免拉低/拉高數字造成誤解）。
- **首次正確決策時間**：個體第一次「依現有資訊（局部視野 BFS 或系統建議）做出有依據移動」的 tick，用來衡量系統能多快讓人「做出對的判斷」，而不只是最終有沒有活下來。
- **誤入危險路線比例**：個體依當下已知資訊移動後，卻仍走進煙霧濃度 > 0.7 的格子的比例——代表「資訊落後於現場實際狀況」的情形有多常發生。
- **改道成功率**：原定樓梯間/通道路線在移動前失效（被火封住）而被迫重新規劃的人之中，最終仍成功逃脫的比例。分母只算「曾經被迫改道過的人」，不是全體人數。
- **弱勢對象辨識率**：行動不便者與年長者之中，系統成功辨識並標記為優先協助對象的比例。

以上五項若在某次模擬中完全沒有相關樣本（例如沒人改道、沒有弱勢對象角色），會標示為 **N/A**，不會硬算成 0 而扭曲平均值；批次彙總時也只對「有樣本」的場次取平均。

### 系統獲知受困位置平均時間

新增追蹤「系統端最早得知某人受困/卡住位置」的時間點，並計算平均值。這個時間點只在該人**手機仍有電且訊號未斷**時才會被記錄；只統計「最終沒有成功逃脫（死亡或受困）」的人，已逃脫者中途卡過也不列入計算——目的是衡量「系統能多快知道『這個人現在有麻煩』」，而非單純的存活率。

> 註：目前程式裡「系統端」與「消防/救援端」是同一個概念（人機互動迴圈已簡化為「系統提供資訊，個體依此自行決策」，尚未做出獨立的消防隊/119物件），此指標可理解為「對外求救管道得知受困位置的平均時間」。

### 人物屬性模型大幅擴充

`PersonProfile` 現在不只影響速度與 CO 耐受度，而是真正驅動不同的行為邏輯：

- **NORMAL_SOLO / STAFF / CUSTOMER**：一般成年人參數；其中 **STAFF** 對場域熟悉，即使沒有直接視線也知道本樓層所有樓梯間（備用逃生動線）位置，CUSTOMER 則沒有這個加成。
- **WITH_CHILD**：一開始會花數個 tick 尋找/確認小孩安全，這段期間不會主動逃生；移動速度較慢。
- **IMPAIRED（行動不便）/ ELDERLY（年長者）**：CO 耐受閾值較低、恐慌易感度較高；行動不便者每個 tick 有機率選擇「原地等待救援」而非自行冒險移動。

### 恐慌與服從機制

- 每個人有一個隨所在位置煙霧濃度動態調整的 `panicLevel`（0=冷靜 ~ 1=極度恐慌）。
- 恐慌程度越高，該 tick「愣住不動」與「誤操作/不服從系統建議、隨便亂走」的機率越高（各自有獨立的係數）。
- 新增「**服從率**」機制：即使系統給出建議路線，個體收到建議後仍只有一定機率會真的照做，其餘時間依自身直覺行動——模擬真實情境中人不會百分之百信任/服從系統的狀況。

### 同行者與批次失效變數

- **同行者**：兩人可被設定為同行，當同伴落後太多時，個體可能選擇等待或折返確認同伴狀況。
- **手機沒電**：每個人初始就有一定機率手機沒電/沒帶在身上；模擬過程中緊張操作也可能導致電力耗盡，沒電後完全無法再跟系統互動。
- **網路/節點斷線**：獨立於手機電力之外，每個 tick 也有機率訊號或系統節點斷線，導致暫時失去連線。
- **定位誤差**：手機/系統回報的位置帶有常態分布誤差，只影響系統「看到」的位置，不影響個體實際移動與判定邏輯。

這些變數共同決定「有系統」情境下，系統建議實際上能不能送達、送達的位置準不準——「有系統」不再等於「萬能且零延遲的系統」。

### 防火門（FireDoor）邏輯接上火勢蔓延

防火門若確實關好，能有效阻絕火勢延燒；若沒關好（`isOpen`），行為會退化成跟普通門一樣，擋不住火也可能被燒到卡死。這個屬性現在真正影響火勢蔓延的判定，不再只是預留欄位。

### 系統判斷準確率計算邏輯（延續前次修正）

準確率的計算比較兩次模擬中每個人的**實際結局**（逃脫 / 死亡 / 模擬結束時仍受困），只有「無系統時逃脫、但有系統時沒有逃脫」才算一次錯誤決策；「系統救活了原本會死的人」不算錯誤，也不會反過來加分——這是保守的錯誤率定義，只抓「系統讓結果變差」的情況。

### 感測器誤差與隨機故障機制（延續前次修正）

- 感測器讀數每個 tick 帶有常態分布雜訊，可能誤報（偏高）也可能漏報（偏低），危險判定每個 tick 重新計算。
- 除了「被火燒到即物理損毀」外，感測器也有獨立的每 tick 隨機故障機率（跟現場是否有火無關），故障後不再更新讀數，並被系統保守視為該處已經著火。

## 尚未實作 / 已知範圍限制

- **人機互動迴圈**：目前是純批次自動模擬（系統提供資訊、個體依規則與服從率內部計算決策），沒有真人可即時介入、回報現場狀況（例如「西側走廊已阻塞」）並回饋更新地圖的互動機制。
- **情境設計者輸入介面**：地圖、起火點、人數目前全數隨機生成，沒有讓使用者手動指定「哪些設備故障/哪些出口已阻塞」的介面（批次模擬模式本身也不需要這個功能）。
- **體驗模式**：尚未提供讓真人扮演主角、逐步查看系統介面與建議的互動式體驗模式，目前僅有批次模擬模式。

### 煙霧擴散邏輯改進

## 簡短說明

- Fireeeee 是一套以 Java 實作的火災逃生模擬器，用來評估「沒有偵測/資訊系統（無系統）」與「部署偵測器並分享資訊的智慧支援系統（有系統）」在火災環境下的逃生效率差異。
- 本專案以模擬結果量化系統效益，並提供可重複、可參數化的大量蒙地卡羅模擬平台（用以驗證系統設計假說）。

## 重點結論（數值總結）

> ⚠️ 以下數值來自本次量化指標與人物行為模型大幅擴充**之前**的舊版模擬結果。由於準確率定義、存活時長記錄方式、以及人物行為（恐慌/服從率/連線失效等）都已改變，舊數字已經不能反映目前版本的行為，需要重新跑一輪模擬才能更新這裡的結論。

- 無系統情況的平均逃生率：約 3.33%。
- 有系統情況的平均逃生率：約 62.96%。
- 系統對「每位個體平均存活時長」的優化程度與火勢蔓延速率成正比，實驗得到的比例常數約為 31.705（即在本模型與參數下，火勢速率增加時，系統的優化效益也隨之提升）。

## 模擬方式（方法說明）

- **比較目標**: 在同樣「沒有消防車、沒有滅火器、火勢按固定規則蔓延」的情況下，比較有無系統下「存活率」「存活時間」以及各項暴露/決策品質指標的差異。
- **關鍵觀察變數**: 本模擬把系統效益視為「使用者會知道哪些資訊（感測器回報/全域共享）」，並依此影響每一步中個體的決策（局部視野 vs 全域訊息），同時疊加恐慌、服從率、連線/定位失效等人為變數的影響。
- **每次模擬流程**:
  1. 隨機產生建築（樓層、牆、門、防火門、樓梯、單一出口）。
  2. 放置火源（單一格）並啟動火/煙擴散規則。
  3. 放置受難者，並隨機賦予其人物屬性（NORMAL_SOLO / WITH_CHILD / IMPAIRED / ELDERLY / STAFF / CUSTOMER）。
  4. 兩組實驗：一組為「無系統」行為（局部視野、遇濃煙視野受限、遇煙採隨機移動等）；另一組為「有系統」行為（偵測器佈署、偵測器資訊全體共享、依服從率決定是否採納建議）。
  5. 以 tick 為單位運行至建築「被火焚毀（所有非牆格著火）」或所有人皆逃生/死亡/模擬時間上限為止。

## 模擬假設與缺陷（重要限制）

- **無外部滅火資源**: 模擬中不含消防車、滅火器、外部救援的滅火行為。
- **火勢行為**: 火以固定規則在每個 tick 向四方向擴散（每隔 tick 擴散）；不模擬風或複雜熱流；防火門若確實關好可阻絕延燒。
- **煙霧行為**: 見 2026-07-03 更新內容。
- **偵測器規則**: 平均每約 3 平方格放一個 detector。
  - 若偵測器所在格著火，該偵測器物理損毀（壞掉）。
  - 除此之外，每個 tick 也有一個獨立的小機率會隨機故障（跟火無關），故障後同樣視為壞掉。
  - 未故障時，偵測器每個 tick 讀到的煙霧濃度會帶有隨機誤差（可能偏高、可能偏低），若「讀到」的濃度大於 0.7 才觸發危險警示——因此系統看到的「危險」不保證等於現場真實狀況。
- **人為失效變數**: 服從率（收到建議不一定照做）、手機沒電、網路/節點斷線、定位誤差，皆會讓「有系統」的建議無法百分之百送達或精準送達。
- **行為模型簡化**: 人類行為以啟發式（局部 BFS、隨機逃逸、或基於偵測器的路徑搜尋）實作；恐慌、同行者等待、行動不便者待援等僅用簡化規則模擬，沒有真正的群體心理模型，也沒有真人即時互動迴圈。
- **以上假設會影響絕對數值，但對「有系統 vs 無系統」的相對差異仍具指標性。**

## 量化方式（指標定義）

- **逃生率（Survival rate）** = 生還人數 / 總人數。
- **個體存活時長定義**:
  - 若個體逃出：存活時長 = 從起火到該個體**實際逃出**的 tick。
  - 若個體被火燒死或CO中毒死亡：存活時長 = 從起火到該個體死亡的 tick。
  - 若模擬結束時個體仍受困（既沒逃出也沒死）：存活時長以模擬結束當下的 tick 記錄，僅供參考，判斷指標不會把這種人跟「逃出」的人混為一談。
- **系統判斷準確率（System Mistake Rate）**: 只有「無系統時逃脫、有系統時沒有逃脫（死亡或受困）」才算一次錯誤判斷；「有系統救活了無系統會死的人」不算錯誤，也不會反過來加分。
- **進入高煙區比例**：個體曾經處於煙霧濃度 > 0.7 的格子的比例。
- **臨界CO暴露比例**：個體累積CO量曾經達到其個人耐受閾值 80% 的比例。
- **誤入危險路線比例**：依現有資訊移動卻仍走進高煙區的比例。
- **改道成功率**：曾被迫重新規劃路線的人之中，最終成功逃脫的比例（無樣本時為 N/A）。
- **弱勢對象辨識率**：行動不便者/年長者中，被系統成功辨識標記的比例（無樣本時為 N/A）。
- **平均逃脫所需時間**：成功逃脫者的平均逃出 tick。
- **平均首次正確決策時間**：個體平均第一次做出有依據決策的 tick。
- **系統獲知受困位置平均時間**：對最終未逃脫成功者，系統平均最早得知其受困位置的 tick（無樣本時為 N/A）。
- **系統優化值（本次報告呈現）**:
  - 經觀察，在不同的無系統生還率下，有系統的生還率與無系統生還率普遍趨近於 `y=f(m)(x+k)` 的常數差距關係，其中，`k`為常數，代表有系統對初始無系統情況改善的量，`f(m)`則是不同樓型對生還率的影響常數，`f(m)`並非單一參數的函數，而是簡化多個影響因素(例如人數、樓層數、樓長與寬、火源位置、門口位置等等)後的表示。
  - 對每個人計算 (存活時長_with_system − 存活時長_without_system)，求全體個體的平均值。報告中指出此平均值隨火勢蔓延速率變化成正比，比例常數為 31.705。

## 量化結果（本次模擬摘要）

> ⚠️ 同上方提醒，以下數字是本次擴充前的舊版結果，僅供參考，尚待重新模擬更新。

在所選的測試集（多次隨機場景，從小套房到十層百貨公司大樓）與模型參數下：

- 無系統的整體平均逃生率：約 3.33%（代表沒有偵測器/資訊支援時的極低逃生機率）。
- 有系統的整體平均逃生率：約 62.96%（部署偵測器並共享資訊後，逃生率顯著上升）。
- 系統對「每位個體平均存活時長」的優化量不是線性增長，而傾向於一個與火勢速率成比例的常數尺度（本實驗得 31.705）。
- 有系統在小樓型中的準確率為97%至98.5%，優化常數約為9.44，在大樓型中，因為運行時間過長，僅透過百次隨機模擬與小樓型數據，估計準確率較低，優化常數則較高（在單筆大樓型的測試資料中，優化常數達96.471，細節見報告文檔），代表在大樓型中，系統相對於小樓型，更傾向讓人積極地躲避火源。

## 如何執行模擬（快速開始）

### 1. 編譯

以專案根目錄為工作目錄：

```bash
javac -d out src/simulator/*.java
```

### 2. 執行

執行主程式（在 sim.main 可調參數）：

```bash
java -cp out simulator.sim
```

在 main 中，可透過呼叫 `s.work(iterations, new Range(Hmin,Hmax), new Range(Rmin,Rmax), new Range(Cmin,Cmax))` 調整模擬次數與地圖尺寸範圍。

> 注意：目前 `main()` 內預設呼叫是 `s.work(1, new Range(1,11), new Range(5,51), new Range(5,51))`，僅跑 1 次迭代，適合快速測試；正式蒐集數據時請自行調高 `iterations`。時間複雜度會隨房子體積（樓高 × 樓寬 × 樓長）大幅上升，不建議單一場景超過 1,000,000 立方單位。

### 3. 輸出檔案

- 每次 run 會產生兩個過程檔案：`defaultResult.txt`（無系統過程）與 `smartResult.txt`（有系統過程）。僅會保留最後一次模擬的過程檔案。
- 統計結果（生還率、系統嚴重失誤機率、進入高煙區比例、臨界CO暴露比例、誤入危險路線比例、改道成功率、弱勢對象辨識率、平均逃脫所需時間、平均首次正確決策時間、系統獲知受困位置平均時間）會輸出到標準錯誤（stderr），便於在自動化批次實驗中蒐集；無樣本的指標會標示為 N/A。

## 解讀建議與局限

- 本模擬提供「比較性」而非「絕對值」的證據：絕對逃生率會依建築配置、人口密度、火源位置、門/牆分佈與偵測器密度顯著改變；但在相同假設下的「有系統 vs 無系統」相對差異仍具參考價值。
- 模型簡化了真實世界物理（風、溫度等）與群體心理，未包含主動救援隊伍與外部滅火，也尚未實作真人即時互動迴圈與情境設計者輸入介面，這些會影響實際部署的數值結果。
- 「比例常數 31.705」是本模型在所用參數與火勢規則下的實驗性結果，並非普適常數；若改變火勢擴散速率、偵測器密度或人物行為參數（恐慌係數、服從率等），此常數會變動。建議做敏感度分析。
- 感測器具備讀數誤差與隨機故障機率，個體端也具備服從率、手機/網路失效、定位誤差等變數，這代表「有系統」不再是永遠正確、永遠送達的資訊來源——系統判斷準確率與逃生率會因此更貼近真實部署情境，但也代表更早期版本（零誤差、零失效）的數值已經不能直接沿用。
